### Post-Demo: Critical SMTP Authentication Discovery

- Attempted to configure the custom MCP email server with the manager's corporate Outlook account (`@apollo-gs.com`). Navigated to Microsoft's security settings (`mysignins.microsoft.com`) to generate an App Password but discovered the option was entirely absent from the available authentication methods.
- **Root cause identified:** The corporate Microsoft 365 tenant has disabled basic SMTP authentication (App Passwords) organization-wide in favor of Modern Authentication (OAuth2). This is a Microsoft-enforced security policy — not a configuration error — confirmed by Microsoft's own documentation stating "Authentication Method: OAuth2/Modern Authentication" for all Outlook SMTP connections.
- Conducted a multi-session architectural review using Claude as a senior technical sounding board to determine whether this was a solvable problem or a hard platform constraint. Claude confirmed unequivocally: *"There is no password-based way into her mailbox left."* Any application sending email through a corporate Outlook account — whether a custom script, Zapier, or Anthropic's own official connector — requires an Azure App Registration with admin consent. This is Microsoft's tenant security model, not a limitation of our code.

### Strategic Decision: Architecture Pivot to Power Automate (Option 2)

- Developed a two-option proposal for management:
  - **Option 1:** Request an Azure App Registration from IT (delegated `SMTP.Send` + `offline_access` permissions) to maintain the custom MCP automation pipeline. Advantage: preserves the Claude-driven workflow. Disadvantage: requires IT admin involvement, formal review process, and ongoing maintenance of OAuth token refresh.
  - **Option 2:** Rebuild the workflow entirely within Microsoft Power Automate using the pre-consented Outlook connector (`Send an email V2`), with Microsoft Lists as the employee database. Advantage: zero IT dependency, runs on infrastructure already approved by the organization. Claude's role transitions from "email executor" to "coding assistant and template architect."
- Drafted a professional communication presenting both options transparently to the manager. Received directive to proceed with Option 2 as the immediate deployment path.
- Began preliminary research into Power Automate flow architecture: scheduled cloud flows, SharePoint connector for Microsoft Lists, OData filter queries, and Outlook V2 email action with raw HTML body injection via Compose actions.

## Day 18: Power Automate Implementation and End-to-End Validation
**Date:** July 17, 2026 (Friday)

### HTML Template Optimization for Outlook Compatibility

- Leveraged Claude as a senior email-development consultant to audit the custom HTML onboarding template for cross-client rendering. Key findings and fixes applied:
  - Added `width:100% !important; height:100% !important;` directly to the `<body>` inline style as insurance against corporate mail gateways that strip `<head><style>` blocks entirely.
  - Replaced the `favicon.ico` logo source (a browser-tab icon format, not an image asset) with a proper `logo.png` reference to prevent blurriness and rendering failures in mail clients that don't support `.ico`.
  - Confirmed that the existing `@media` queries and `mso-` reset properties must remain in the `<style>` block (cannot be inlined by definition) and are correctly placed.
  - Documented the known cosmetic limitation: `border-radius` on header, footer, day badges, and contact cards will render as square corners in classic desktop Outlook (Word rendering engine limitation) but display correctly in new Outlook, Outlook on the web, and all other major clients. Determined that building VML workarounds for this edge case is not justified for an internal HR email.

### Microsoft Lists Database Setup

- Created the `Employee Onboarding Tracker` list on SharePoint with precisely named columns (no spaces in internal names to avoid Power Automate's `_x0020_` encoding issues):
  - `Name` (renamed from default "Title" column) — Single line of text
  - `Role` — Single line of text
  - `Department` — Single line of text
  - `Manager` — Single line of text
  - `StartDate` — Date only (no time picker, to avoid off-by-hours date comparison bugs)
  - `NewHireEmail` — Single line of text (deliberately not "Person" type, since new hires may not yet exist in the tenant)
  - `Day1Sent`, `Day3Sent`, `Week1Sent` — Yes/No (default: No)
  - Added a test row with today's date for flow validation.

### Power Automate Flow Construction

- Built a `Scheduled cloud flow` named "Onboarding Email Sequence" with the following architecture:
  1. **Recurrence trigger:** Runs daily at 08:00 with explicit time zone setting (not UTC default) to ensure accurate date comparisons.
  2. **Get items (SharePoint):** Connected to the Microsoft Lists tracker with OData filter `Week1Sent eq 0` — a single filter that efficiently pulls every row with at least one pending email stage.
  3. **Apply to each loop:** Iterates over all pending employee records.
  4. **Three Condition branches** (Day 1 / Day 3 / Week 1): Each uses a "catch-up" date logic (`greaterOrEquals` + `addDays`) rather than strict same-day equality — a deliberate engineering decision ensuring that if the flow misses a run (weekend, outage, editing), it automatically catches up on the next execution instead of silently skipping that employee's email forever.
  5. **Compose action → Send an email (V2):** Raw HTML template placed in a Compose action with dynamic content from the List mapped to `{{NAME}}`, `{{ROLE}}`, `{{DEPARTMENT}}`, etc., then referenced by the Send Email action's Body field — sidestepping Power Automate's rich-text editor which would otherwise mangle the HTML.
  6. **Update item:** After each successful email send, flips the corresponding boolean (`Day1Sent`, `Day3Sent`, or `Week1Sent`) to `true` to prevent duplicate sends.
